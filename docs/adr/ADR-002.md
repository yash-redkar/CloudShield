# ADR-002: Use Terraform for Infrastructure as Code

- **Status:** Accepted
- **Date:** 2026-07-01

## Context
CloudShield needs infrastructure to be reproducible and version-controlled rather than manually configured in the AWS Console. Options considered: Terraform, AWS CloudFormation, manual console configuration.

## Decision
Use Terraform to define and provision all AWS infrastructure.

## Alternatives Considered
- **AWS CloudFormation** — native to AWS with no extra tooling, but locks the platform to AWS-only syntax and tooling, and has historically slower iteration (nested stack complexity, YAML verbosity) compared to Terraform's module system.
- **Manual AWS Console configuration** — fastest for a one-off resource, but not reproducible, not reviewable via pull request, and prone to drift since changes aren't tracked in code.

## Consequences
- Infrastructure changes go through code review (pull requests) instead of console clicks.
- Requires managing Terraform state (addressed separately — remote backend in S3 with locking).
- Terraform's provider-agnostic design leaves room to extend beyond AWS later, though not a near-term goal.
