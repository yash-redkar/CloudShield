# ADR-003: Use k3s over EKS and Docker Compose

- **Status:** Accepted
- **Date:** 2026-07-01

## Context
CloudShield needs a container orchestration layer that can run and manage multiple applications reliably. Options considered: AWS EKS (managed Kubernetes), k3s (lightweight Kubernetes) on EC2, and Docker Compose.

## Decision
Use k3s running on a single EC2 instance for v1.

## Alternatives Considered
- **AWS EKS** — production-grade managed Kubernetes, but adds control-plane cost (~$0.10/hr) and setup complexity (IAM roles for service accounts, VPC CNI configuration) that isn't necessary to prove the platform's core config-driven deployment model.
- **Docker Compose** — simplest option and sufficient for running containers locally, but lacks orchestration features (self-healing, rolling updates, declarative scaling) that the platform is meant to demonstrate.

## Consequences
- Faster to stand up v1 and iterate on config-driven deployment logic without AWS-specific Kubernetes complexity.
- k3s is a lightweight but conformant Kubernetes distribution, so manifests and concepts (Deployments, Services, ConfigMaps) transfer directly to EKS if migrated later.
- A future migration to EKS remains possible and would be tracked as a new ADR if pursued, rather than assumed as part of this decision.
