# ADR-001: Build CloudShield as an Internal Developer Platform

- **Status:** Accepted
- **Date:** 2026-07-01

## Context
Deploying an application to production typically requires knowledge across many domains: cloud infrastructure, networking, containers, Kubernetes, CI/CD, monitoring, and security. Repeating this process manually for every application leads to inconsistent environments, configuration drift, and slow onboarding.

## Decision
Build CloudShield as a reusable Internal Developer Platform (IDP Lite). Application teams interact with the platform through a simple configuration file (`apps/*.yaml`) instead of managing infrastructure directly.

## Alternatives Considered
- **Per-application deploy scripts** — fastest to write initially, but logic duplicates across every app and drifts over time as each script is maintained independently.
- **Manual deployment (console/SSH per release)** — no upfront engineering cost, but doesn't scale past one app and leaves no audit trail or repeatability.

## Consequences
- Higher upfront engineering effort to build the platform before any app benefits from it.
- Once built, adding a new application requires only a config file, not new infrastructure code.
- The platform itself becomes a maintenance responsibility, separate from any single application.
